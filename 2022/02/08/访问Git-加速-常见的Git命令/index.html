仓库下载加速

将原始Repo链接之中的 “ https://github.com” 替换成对应的镜像网站

  • https://github.com/repo → https://github.com.cnpmjs.org/repo
  • https://github.com/repo → https://hub.fastgit.org/repo
  • https://github.com/repo → https://gitclone.com/github.com/repo

举例： 我们可以将kubenetes项目clone到本地的时候

  • git clone https://github.com/kubernetes/kubernetes.git

改为以下三种皆可以

  • git clone https://github.com.cnpmjs.org/kubernetes/kubernetes.git
  • git clone https://hub.fastgit.org/kubernetes/kubernetes.git
  • git clone https://gitclone.com/github.com/kubernetes/kubernetes

文件下载加速

操作方法和上面同理

举例： 原地址

  • wget https://github.com/goharbor/harbor/releases/download/v2.0.2/harbor-offline-installer-v2.0.2.tgz

改为皆可以

  • wget https://hub.fastgit.org/goharbor/harbor/releases/download/v2.0.2/harbor-offline-installer-v2.0.2.tgz

免手动替换方法(不建议)

设置全局替换

  • git config –global url.”https://hub.fastgit.org“.insteadOf https://github.co

查看git配置信息

  • git config –global –list

取消设置

  • git config –global –unset url.https://github.com/.insteadof

Git Sheet

原则

  • 每次Git提交代码之前，一定要进行功能测试。并且留下详细的注释， 测试的接口， 本地测试历史。提交自己的代码一定要站在合作者的角度，方便其他人员进行对接（团队项目合作的事情：方便他人就是方便自己）。
  • 要使用Git经常性的提交，原则是一个bug对应一次提交， 一个功能对应一次提交。
  • 有机会一定要使用分支的功能（ Git）的灵魂之一，否则第一次中间会踩很多很多的坑orz…

分支与标签 (checkout & branch)

  • 列出所有现有分支
  • 切换分支
  • 根据当前的头部创建一个新分支
  • 基于远程分支创建新的跟踪分支
  • 删除本地分支

提交标签

合并与重置(merge & rebase)

  • 命令使用原则
    • 如果你想要一个干净的，没有merge commit的线性历史树，那么你应该选择git rebase。 如果你想保留完整的历史记录，并且想要避免重写commit history的风险，你应该选择使用git merge。
    • rebase的本质是寻找两个链表的最近公共节点（每个分支是一个链表，每一次提交是一个节点），采用rebase会破坏一些commit 历史。所以，不要在公共的分支之上使用rebase这个命令合并master
  • 将分支 合并到当前的头部
  • 将当前的头重新定位到<分支> (不要重新发布已发布的提交！)
  • 中止重基
  • 解决冲突后继续重基
  • 使用配置的合并工具解决冲突
  • 使用编辑器手动解决冲突，并(在解决后)将文件标记为“已解决”。

更新和发布(fetch & pull)

  • 显示有关远程
  • 添加名为< Remote >的新远程存储库
  • 从< Remote >下载所有更改，但不要集成到Head中
  • ⚠️ （注意pull 和 fetch的区别）下载更改并直接合并/集成到头中
  • 在远程上发布本地更改
  • 删除远程上的分支
  • 发布标签

记录与撤销 ( log & reset)

  • 显示所有提交，从最新开始
  • 显示特定文件随时间的变化
  • 谁在< file >中更改了内容和时间？
  • 放弃工作目录中的所有本地更改。
  • 还原一个提交(通过产生一个新的具有相反更改的提交)
  • 将head重置为上一次提交, 并放弃自那以后的所有变化)(—soft 和 —hard 的区别)
  • 将head重置为上一次提交，并将所有更改保留为未分阶段的更改。
  • 将head重置为上一次提交， 并保存未提交的本地更改。
